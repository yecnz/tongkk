# 전체 흐름
- 데이터웨어하우스(DW)의 등장 배경과 기존 정보 시스템의 문제점 파악
- 데이터웨어하우스의 정의, 핵심 특성 및 데이터베이스(DB)와의 차이 이해
- 데이터웨어하우스의 시스템 구조(3계층) 및 구축 방식(상향식, 하향식, 절충식) 학습
- 다차원 모델링(MDDB)과 관계형 모델링(스타/스노우플레이크 스키마)의 구조 및 연산 이해
- OLAP의 정의, 특성 및 주요 인터페이스(연산) 정리
- 분산 데이터베이스, 객체지향, 객체-관계, 멀티미디어, 공간 데이터베이스 등 다양한 데이터베이스 응용 분야 학습

# 섹션별 핵심 정리

## 1. 데이터웨어하우스 개념
- **핵심 정의**
    - 의사결정을 위한 효율적인 분석 기반을 제공하는 통합 데이터 저장소
    - 이질적인 데이터의 추출, 변            변환, 통합 과정을 통해 구축된 거대한 분석 전용 데이터 창고
- **핵심 설명**
    - **출현 배경**: 정보의 위기(information crisis) — 방대한 데이터에 비해 의사결정 활용 가능한 정보은 매우 제한적임(풍요 속의 빈곤, 거미줄 구조).
    - **기존 정보 시스템의 문제점**: 데이터의 비통합성, 비일관성, 비신뢰성, 단기성.
- **반드시 기억할 세부 항목 (기존 시스템의 문제점)**
    - **데이터의 비통합성**: 여러 업체에 의해 서로 다른 시기에 개발되어 데이터가 통합되어 있지 않음.
    - **데이터의 비일관성**: 업무 프로그램 중심 개발로 인한 데이터 중록 및 표현/의미상의 차이 발생.
    - **데이터의 비신뢰성**: 데이터 생성 주기나 추출 알고리즘 차이로 인한 결과값 불일치.
    - **데이터의 단기성**: 현 시점 위주의 단기간 데이터 위주이며, 과거 이력 데이터 부족으로 장기 추이 분석 어려움.

## 1.2 데이터웨어하우스의 정의 및 특성
- **핵심 정의**
    - 의사결정에 필요한 올바른 정보(right information)를 올바른 형식(right form)으로 적시(right time)에 제공하기 위한 저장소.
- **데이터웨어하우스의 특성**
    - **주제 지향성(subject-oriented)**: 분석 프로그램 중심의 데이터 구조로, 분석 주제별로 요약됨.
    - **통합성(integrated)**: 외부 문서(웹, 엑셀, 텍스트 등)와의 통합 및 통일된 형식으로의 변환 필요 (가장 중요한 특성).
    - **시계열성(time-variant)**: 과거부터 현재까지의 스냅샷(snapshot) 데이터를 축적하여 추이 분석 가능.
    - **비휘발성(nonvolatile)**: 수정이나 삭제가 거의 없는 검색/추가/적재 중심의 데이터베이스.

## 1.3 데이터웨어하우스 시스템
- **핵ess 구조: 3-계층 구조(3-tier architecture)**
    - **원본(source) 데이터 계층**: 운영 데이터, 비정형 데이터, 외부 데이터 포함.
    - **데이터 저장소 계층**: 핵심 계층. 관계형 DB 또는 다차원 DB 사용. 
        - **메타데이터 저장소**: 데이터 모델, 변환 규칙, 원본/대상 DB 설명 등을 저장.
    - **클라이언트 계층**: 분석 도구 포함 (보고서 도구, OLAP 도구, 데이터 마이닝 도구).
- **데이터웨어하우스 구축 방식**
    - **상향식**: 소규모 데이터 마트(data mart)를 먼저 구축 후 전사적 데이터웨어하우스(EDW)로 통합.
    - **하향식**: 전사적 데이터웨어하우스를 먼저 구축 후 필요에 따라 데이터 마트를 추가.
    - **절충식**: 기획/설계는 하향식, 실제 구축은 상향식(중요 분야부터)으로 진행.
- **데이터 마트(data mart)**
    - 조직의 일부 사용자(비즈니스 부서 등)를 위해 주제 범위를 제한하여 구축한 '미니 데이터웨어하우스'.

## 2. 데이터웨어하우스 설계
- **핵심 설명**
    - **다차원성**: 다양한 분석 관점 간의 조합을 통해 데이터 상호 연관성 분석 가능.
    - **다차원 모델링(multi-dimensional modeling)**: 데이터를 다차원성을 유지하도록 구조화함.
- **다차원 모델링 방법**
    - **다차원 데이터베이스(MDDB)**: 데이터 큐브(data cube) 개념 이용.
    - **관계형 데이터베이스**: 스타 스키마(star schema) 개념 이용.

### 2.1 다차원 모델링 (MDDB)
- **핵심 정의**: 관계형 DB의 한계를 극복하기 위해 개발된 분석 전용 DB로, 큐브 구조를 가짐.
- **구조적 요소**
    - **큐브(cube)**: 셀(cell) 형태의 저장소. 차원 멤버들의 모든 가능한 조합 수만큼 셀이 존재.
    - **차원(dimension)**: 데이터에 대한 하나의 분석 관점/시각.
    - **차원 멤버(dimension member)**: 큐브의 축 좌표.
    - **측정값(measure)**: 큐브 셀 안에 저장되는 값.
- **다차원 질의(multi-dimensional query) 연산**
    - **슬라이싱(slicing)**: 큐브를 절단하여 특정 단면을 분석.
    - **다이싱(dicing)**: 큐브의 특정 부분 영역을 선택하여 분석.
    - **피보팅(pivoting)**: 큐브의 축 위치를 변경.
- **다차원 모델 요소**
    - **데이터 큐브(data cube)**: n-차원 큐브 구조로 모델링된 기본 구조.
    - **개념 계층(concept hierarchy)**: 차원 내 수준 간의 수직적 상하 관계 (계층적 조직화).

### 2.2 관계형 모델링
- **스타 스키마(star schema)**
    - **정의**: 하나의 사실 테이블(fact table)을 중심으로 여러 차원 테이블(dimension table)이 연결된 방사성 구조.
    - **구성 요소**:
        - **사실(fact)**: 측정값(measure) 또는 집계값(수치 데이터: 판매량, 매출액 등).
        - **차원(dimension)**: 사실을 바라보는 관점(대리점명, 제품명 등).
    - **장점**: 구조가 단순하고 속도가 빠름.
    - **단점**: 차원 테이블이 비정규화되어 데이터 중복이 많음.
    - **스타 조인(star join)**: 일대다 참조 관계를 이용하여 사실 테이블과 차원 테이블을 결합하는 연산.
- **스노우플레이크 스키마(snowflake schema)**
    - **정의**: 데이터 중복을 줄이기 위해 차원 테이블을 정규화한 형태.
    - **특징**: 차원 테이블이 분리되어 구조가 복잡하고 SQL문이 복잡해짐.
- **은하수 스키마(galaxy schema)**
    - **정의**: 여러 개의 스타 스키마가 모인 구조로, 같은 차원 테이블을 공유하는 여러 사실 테이블을 포함함. (스타 패밀리 또는 성운 스기마라고도 함).

## 3. OLAP (On Line Analytical Processing)
- **핵심 정의**
    - 사용자가 대규모 다차원 데이터를 온라인에서 동적으로 분석하고 보고서를 만드는 과정.
    - FASMI (Fast Analysis of Shared Multi-dimensional Information)라고도 함.
- **OLAP의 특성**
    - **다차원성**: 주제별 다양한 관점 접근 (가장 중요).
    - **직접성**: 개발자 도움 없이 사용자가 스스로 분석.
    - **대화성**: 반복적 질문을 통한 정보 발견(대화식 인터페이스).
    - **비정형성**: 자유로운 보고서 생성.
    - **자동성**: 메타데이터를 활용한 자동 보고서 생성.
- **OLAP 인터페이스 (주요 연산)**
    1. **드릴-다운(drill-down)**: 요약된 수준에서 상세한 하위 수준으로 분석.
    2. **롤-업(roll-up)**: 상세한 수준에서 요약된 상위 수준으로 분석 (드릴-업이라고도 함).
    3. **슬라이스와 다이스(slice & dice)**: 큐브의 일부분을 잘라 작은 부분 큐브 생성.
    4. **피봇(pivot)**: 분석 중인 데이터의 축을 변경.
    5. **드릴-쓰루(drill-through)**: 요약된 수준에서 연관된 상세 데이터로 이동하여 분석.

## 4. 데이터베이스 응용
### 4.1 분산 데이터베이스
- **정의**: 물리적으로 분산된 DB를 네트워크로 연결하여 하나의 DB처럼 사용하도록 관리하는 시스템.
- **분산 데이터 독립성 (5가지 투명성)**
    - **위치 투명성(location transparency)**: 물리적 저장 위치를 알 필요 없음.
    차원: **중복 투명성(replication transparency)**: 데이터 사본 존재 여부에 신경 쓰지 않음.
    - **단편화 투명성(fragmentation transparency)**: 데이터의 분할(fragment)을 알 필요 없음.
    - **병행 투명성(concurrency transparency)**: 동시 실행 시에도 정상적인 결과 보장.
    - **장애 투명성(failure transparency)**: 일부 시스템/통신망 손상 시에도 정상 처리.
- **분산 데이터베이스 스키마 구조**
    - **전역 스키마**: 모든 데이터 구조와 제약 조건 정의.
    - **단편 스키마**: 전역 스키마를 논리적으로 분할한 단편 정의.
    - **할당 스키마**: 각 단편의 물리적 저장 지역 정의.
    - **지역 스키마**: 지역별 데이터 구조와 제약 조건 정의.

### 4.2 ~ 4.5 기타 데이터베이스 응용
- **객체지향 데이터베이스(object-oriented database)**: 객체 단위로 저장/관리. '데이터 유형 불일치' 문제 해결.
- **객체-관계 데이터베이스(ORDBMS)**: 관계형 DB에 객체지향 모델의 장점을 접목한 하이브리드 모델 (SQL3 사용).
- **멀티미디어 데이터베이스**: 텍스트, 그래픽, 오디오, 비디오 등 비정형 데이터를 저장/관리.
    - **시스템 유형**: 파일 시스템 활용, 관계형 DB 활용, BLOB 활용, 객체지향 DB 활용.
- **공간 데이터베이스**: 다차원 공간의 객체와 공간 관계를 관리.
    - **관리 데이터 유형 3가지**: 지리 데이터(geographic), 기하학 데이터(geometric), 공간 속성 데이터(spatial attribute).

# 비교 포인트

| 비교 항목 | 스타 스키마 (Star Schema) | 스노우플레이크 스무키 (Snowflake Schema) |
| :--- | :--- | :--- |
| **정규화 여부** | 비정규화 (차원 테이블 중복 많음) | 정규화 (차원 테이블 분리) |
| **구조 복잡도** | 단순함 | 복잡함 |
| **조인(Join) 성능** | 빠름 (조인 수 적음) | 상대적으로 느림 (조인 수 많음) |
| **SQL 복잡도** | 단순함 | 복잡함 |

| 비교 항목 | 객체지향 데이터베이스 (OODB) | 객체-관계 데이터베이스 (ORDB) |
| :--- | :--- | :--- |
| **핵점 개념** | 객체지향 개념을 DB에 확장 | 관계형 DB에 객체 개념을 추가/확장 |
| **주요 특징** | 응용 프로그램과 데이터 저장의 일관성 | 관계 모델의 단순성 + 객체지향의 확장성 |

# 시험 직전 체크리스트
- [ ] **데이터웨어하우스의 4가지 특성**: 주제 지향성, 통합성, 시계열성, 비휘발성
- [ ] **기존 시스템의 문제점 4가지**: 비통합성, 비일관성, 비신뢰성, 단기성
- [ ] **DW 구축 방식 3가지**: 상향식, 하향식, 절충식
- [ ] **OLAP 5가지 연산**: 드릴-다운, 롤-업, 슬라이스 & 다이스, 피봇, 드릴-쓰루
- [ ] **분산 데이터 투명성 5가지**: 위치, 중복, 단편화, 병행, 장애 투명성
- [ ] **분산 DB 스키마 4가지**: 전역, 단편, 할당, 지역 스키마
- [ ] **MDDB 큐브 연산**: 슬라이싱, 다이싱, 피보팅
- [ ] **멀티미디어 DB 저장 방식 4가지**: 파일 시스템, 관계형 DB, BLOB 활용, 객체지향 DB 활용
- [ ] **공간 데이터 유형 3가지**: 지리, 기하학, 공간 속성 데이터
- [ ] **FASMI**: OLAP의 핵심 요구사항 (신속한 분석)
